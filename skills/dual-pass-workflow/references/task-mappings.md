# Task Mappings

These mappings describe recurring task shapes for dual-pass. The headings are intentionally pattern-based rather than a fixed registry of skills.

When a repo-local skill or a newer skill matches the same shape, reuse the mapping and adapt the packet instead of treating the examples below as a closed list.

In `Suggested asks`, replace placeholders such as `<review-skill>` with the actual domain skill name for the task.

## Code Bug Fix

Primary result:

- The edited source files

Typical artifact directory:

- Existing task-local output folder if one exists
- Otherwise `.agent/<task-id>/`

Verification:

- Repro before fix
- Targeted tests
- Broader regression checks when cheap

Audit emphasis:

- `factual_or_behavior_issue`: bugs, regressions, broken edge cases
- `unsupported_claim_or_assumption`: "fixed" without a passing repro or test
- `missed_requirement`: forgot acceptance criteria, forgot a file, forgot a migration, forgot docs when required
- `clarity_or_structure_issue`: hard-to-maintain patch, unclear naming, weak error handling

## Writing Or Letter Draft

Primary result:

- The requested draft file

Verification:

- Fact check against user materials
- Format check against the requested structure
- Tone and audience check

Audit emphasis:

- `factual_or_behavior_issue`: wrong dates, names, affiliations, claims
- `unsupported_claim_or_assumption`: praise or criticism not grounded in source material
- `missed_requirement`: omitted points, wrong audience, wrong format
- `clarity_or_structure_issue`: awkward flow, repetition, tone mismatch

## Website Or Frontend Edit

Primary result:

- The edited frontend files

Verification:

- Local build
- Targeted visual check
- Responsive and interaction sanity check

Audit emphasis:

- `factual_or_behavior_issue`: broken behavior, accessibility regressions, layout breakage
- `unsupported_claim_or_assumption`: claimed fix without verifying the page
- `missed_requirement`: missing breakpoint, missing state, missing route or component
- `clarity_or_structure_issue`: inconsistent styling, unclear component boundaries, needless complexity

## Structured Review Folder

Common examples:

- `cs-paper-review`
- `cs-meta-review`

Task root:

- `reviews/<paper-id>/`

Primary result:

- `outputs/review.md`

Artifact directory:

- `outputs/`

Sources of truth:

- `review.yaml`
- `intake/requirements.md`
- `intake/extra-text.md`
- `intake/extracted-paper.md`
- any preflight and evidence artifacts already generated by the domain skill

Ordered preflight (blocking steps run before drafting):

- `integrity-check` → `outputs/pdf-safety-check.md` (blocking)
- `extraction-quality` → `outputs/pdf-extraction-check.md` (blocking)
- `packet-consistency` → `outputs/packet-consistency-check.md` (blocking)
- `visual-triage` → `outputs/pdf-visual-triage.md` (advisory)

Intermediate representation:

- `outputs/evidence-map.md` — required before drafting the review

Verification (post-draft):

- `outputs/reasoning-log.md`
- `outputs/reference-check.md`
- `outputs/integrity-note.md` when required

Audit emphasis:

- `factual_or_behavior_issue`: claims not grounded in the paper, page-level misreads, score-text mismatch
- `unsupported_claim_or_assumption`: over-strong phrasing, speculative criticism, invented related work
- `missed_requirement`: omitted venue field, omitted integrity note, omitted key weakness or key strength
- `clarity_or_structure_issue`: generic reviewer prose, weak summary, poor question selection

Suggested asks:

- `Use $dual-pass-workflow and <review-skill> on reviews/<paper-id> in first-pass mode.`
- `Use $dual-pass-workflow and <review-skill> on reviews/<paper-id> in second-pass mode.`

Low-friction rule:

- `workflow.yaml` is optional for review folders. If it is absent, infer the packet from the standard review layout and write workflow notes under `outputs/`.
- `first-pass` is the default. Use `second-pass` only when you actually want an audit or refinement.

## Coordinated Multi-File Proposal Writing

Common examples:

- `nsf-proposal-composer`

Primary result:

- Multi-file coordinated output:
  - `00-project-summary.tex` (summary)
  - `00-project-description.tex` (main narrative)
  - `11-aim1.tex` through `14-aim4.tex` (units)
  - `20-evaluation.tex` (evaluation)

Typical artifact directory:

- `<proposal-dir>/`

Skill roles:

- primary example: `nsf-proposal-composer`
- compliance example: `nsf-proposal-guardrail`
- refinement example: `nsf-thrust-refiner`
- citation example: `nsf-bibref-filler`

Protected regions:

- `% composer:sync-*` regions are writable by composer and second-pass
- `% refiner:body-*` regions are protected from second-pass unless explicitly unlocked

Verification:

- Cross-section consistency (summary vs intro vs aims vs evaluation)
- Guardrail compliance check
- Cite-key validation via `check_cite_keys.py`

Audit emphasis:

- `factual_or_behavior_issue`: claims not backed by the proposal's own aims, inconsistent PI roles across files
- `unsupported_claim_or_assumption`: overclaimed novelty, evaluation promises without matching milestones
- `missed_requirement`: solicitation requirements not addressed, missing BPC or data management when required
- `clarity_or_structure_issue`: stale bridge text, aim progression unclear, proposal family mismatch
- `citation_integrity`: cite keys that exist in bib but do not actually support the claim they are attached to

Suggested asks:

- `Use $dual-pass-workflow and <proposal-skill> on proposals/<proposal-dir> in first-pass mode.`
- `Use $dual-pass-workflow and <proposal-skill> on proposals/<proposal-dir> in second-pass mode focusing on cross-section consistency.`

## Source-To-Deck Conversion

Common examples:

- `paper-to-beamer`

Primary result:

- `decks/<paper-slug>/<paper-slug>-deck.tex`
- `decks/<paper-slug>/presenter-map.md`

Typical artifact directory:

- `decks/<paper-slug>/`

Workspace contract:

- `README.md`, `templates/paper-deck-base.tex`, `theme/beamerthemeuscmoloch.sty`

Intermediate representation:

- Normalized fact table (required before slide writing)
- Content diagnosis (paper type, strongest contribution, best figure)

Verification:

- `latexmk` compile
- Fact table traceability (every number on slides traces to the fact table)
- Slide overflow and visual density check
- Presenter-map consistency with source comments

Audit emphasis:

- `factual_or_behavior_issue`: numbers not matching the paper, fabricated categories, incorrect method description
- `unsupported_claim_or_assumption`: slide claims not grounded in the normalized fact table
- `missed_requirement`: missing contributions slide, missing limitations, broken figure paths
- `clarity_or_structure_issue`: bullet-dominated runs, method block too compressed, inconsistent presentation mode

Suggested asks:

- `Use $dual-pass-workflow and <source-to-deck-skill> on <source-material> in first-pass mode.`
- `Use $dual-pass-workflow and <source-to-deck-skill> on <task-root> in second-pass mode.`

## Multi-Source Deck Assembly

Common examples:

- `deck-assembler`

Primary result:

- `decks/<talk-slug>/<talk-slug>-deck.tex`
- `decks/<talk-slug>/assembly-map.md`

Typical artifact directory:

- `decks/<talk-slug>/`

Workspace contract:

- `README.md`, `templates/paper-deck-base.tex`, `theme/beamerthemeuscmoloch.sty`, at least one existing paper deck

Verification:

- `latexmk` compile of merged deck
- Missing macro or package detection
- Broken figure path detection
- Continuity checks per `references/continuity-checks.md`

Audit emphasis:

- `factual_or_behavior_issue`: broken compilation, missing figures, style drift
- `unsupported_claim_or_assumption`: bridge slides that overclaim connections between papers
- `missed_requirement`: missing talk-overview slide, duplicate wrappers not removed, assembly-map incomplete
- `clarity_or_structure_issue`: throughline unclear, audience assumption inconsistent across segments

Suggested asks:

- `Use $dual-pass-workflow and <deck-assembly-skill> on decks/<talk-slug> in first-pass mode.`
- `Use $dual-pass-workflow and <deck-assembly-skill> on decks/<talk-slug> in second-pass mode.`

## Citation Patching With Support Risk

Common examples:

- `nsf-bibref-filler`

Primary result:

- Patched `.tex` files with verified citations
- Updated `working.bib` with machine-added entries

Verification:

- `check_cite_keys.py` on all touched `.tex` files
- Factual support check: each inserted cite key's paper must actually support the sentence it is attached to, not just topic-match

Audit emphasis:

- `factual_or_behavior_issue`: cite key exists in bib but the referenced paper does not support the claim
- `unsupported_claim_or_assumption`: topic-match used as support, self-citation cluster used for field-wide claims
- `missed_requirement`: visible `\todo{}` not left where uncertainty remains, `working.bib` not updated
- `clarity_or_structure_issue`: citation dumps, inconsistent density across sections

When to use dual-pass:

- Second-pass is especially valuable here because first-pass citation placement carries the highest hallucination risk among the core skills. The second-pass should independently verify that each inserted citation actually supports its claim, not just that the key resolves.

Suggested asks:

- `Use $dual-pass-workflow and <citation-skill> on <task-root> in first-pass mode.`
- `Use $dual-pass-workflow and <citation-skill> on <task-root> in second-pass mode focusing on citation integrity.`
